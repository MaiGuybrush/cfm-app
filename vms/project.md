# 專案目標
建立一個標準化、集中式、且可擴充的通用影像建模平台（以下稱 UMS），為公司各部門提供影像資料上傳、標註、資料管理、模型訓練、模型管理與部署的一站式服務，降低重複建置成本並加速 AI 在業務場景的落地。

## 執行重點（Executive summary）
- **第一階段**：以新專案方式建立 UMS 平台核心，整合現有 AIADJ/廠務系統的功能需求，確保新平台能完全替代現有系統功能。
- **第二階段**：在 UMS 平台穩定運行後，執行與現有系統的整合與遷移，提供逐步遷移（phase-based migration）與回退機制，降低業務中斷風險。
- 階段性執行可確保風險可控，第一階段專注於建立完整可用的平台，第二階段專注於平穩遷移。

## 預期效益
- 統一建模流程，減少重複工作與錯誤率。
- 集中管理計算資源（GPU 叢集），提高資源使用效率與成本可視化。
- 建立模型資產庫（含版本管理、可追溯性），便於重現與審計。
- 降低各部門導入影像 AI 的技術門檻，提供自助式訓練與部署流水線。

## 範圍
In-scope:
- 影像資料上傳與存放（物件儲存介面）
- 標註工具整合（內建或整合第三方）
- 訓練工作流程與 GPU 排程（Job scheduler / Kubernetes / Slurm 等）
- 模型版本管理、模型登錄（Model Registry）與部署介面（推論服務）
- 權限與審計（單點登入整合、角色權限）
- 開放 API，供 ADR/HOP/廠務 等系統串接

Out-of-scope（初期）:
- 現有 AIADJ/廠務 系統的大幅改造（以遷移策略執行）
- 特殊非影像型別之模型訓練（除非有明確需求）

## 主要利害關係人
- 產品負責人：定義需求、驗收標準
- 專案經理：時程與資源管理
- 平台開發團隊：前端、後端、資料工程、DevOps
- AI/ML 團隊：模型訓練、驗證
- 廠務/AIADJ/ADR/HOP 等業務單位：使用者、遷移資料提供者
- IT/資安：帳號、網路、資安與合規

## 功能需求（詳細）
1. 使用者專案管理
	- 專案可由使用者建立、分享給團隊成員，支援專案權限設定（Owner/Editor/Viewer）
	- 專案內可設定預設計算配額、可見資料集與模型範圍
2. 每個專案下包含
	(1) 資料集管理
		 - 建立 / 維護 / 刪除資料集
		 - 支援資料型別：
			  * 圖像 + 單一標注
			  * 圖像 + 多標注
			  * 圖像 + (矩形框 + 標注)
			  * 圖像 + (多邊形框 + 標注)
		 - 資料集版本管理、樣本瀏覽、標註品質檢查、以及 train/val/test 切分工具
	(2) 模型管理
		 - 選擇模型架構與合適的資料集進行建模
		 - 支援增量訓練（fine-tuning / continued training）選項
		 - 支援對既有模型進行重新訓練（retrain）並保留歷史版本與元資料
		 - 提供模型版本控制資訊（版本標籤、建立者、訓練參數、評估指標、artifact 連結）
		 - 提供刪除模型功能以及模型建置 / 訓練狀態查詢 API/介面
	(3) 佈署管理
         - 提供標準Edge端推論程式
		 - 部署設定、流量管理、回滾與監控整合
3. 使用者 / 團隊管理
	- 使用者帳號與團隊（團隊可擁有多個專案）管理
	- 權限模型支援 Owner / Editor / Viewer 與專案層級的存取控制
	- 與公司現有 SSO/LDAP/AD 整合（見第 6 點）
4. API 管理
	- 提供 REST/GRPC API 與 Webhook，支援 API 權限控管與金鑰管理
	- API 使用率（Rate limiting）與稽核日誌
5. GPU 資源管理
	- GPU 叢集/節點的資源配額、排程策略與監控
	- 支援排隊、優先權、資源限制與計費/配額統計報表
	- 支援查詢當前資源使用狀態與歷史紀錄的介面
6. 整合現有 SSO 機制
	- 與公司 SSO/OIDC/LDAP/AD 整合，支援 SSO 登入與單一來源的使用者授權
	- 支援角色映射與自動同步（如有必要），並記錄審計日誌

備註：原先的「監控與可觀察性」、「安全性與合規」等項目保留於非功能性需求與風險控管章節中，必要時在上述各功能點中再補充具體監控與稽核要求。

## 非功能性需求
- 可用性：目標 99.5%（關鍵服務）
- 可擴展性：支援橫向擴充 GPU 與節點
- 安全性：支援公司 SSO 現有單一簽入機制
- 操作性：提供 UI 與 CLI/API，文件與示例
- 可維運性：監控、告警與自動化部署（CI/CD）

## 高階系統架構（建議）
- Web 前端：使用者介面（標註、管理、監控）
- API 層：認證、授權、業務邏輯
- 資料層：物件儲存（影像）、元資料 DB（Postgres / MySQL）
- 標註服務：內建或第三方整合（CVAT、Label-Studio 等）
- 訓練平台：Kubernetes + GPU 節點 or 專用 Slurm 叢集
- 模型註冊與倉儲：Model Registry（如 MLflow / Harbor for images）
- 推論服務：K8s / KFServing / custom microservices
- 監控堆疊：Prometheus + Grafana、Elastic Stack（Log）
- 身份與存取：公司 SSO / OIDC、RBAC

## 階段性執行策略

### 第一階段：UMS 平台建立
**目標**：建立完整且獨立運作的 UMS 平台，涵蓋現有系統（AIADJ/廠務）所需的核心功能

**執行重點**：
- 以新專案方式開發，不直接修改現有 AIADJ/廠務系統
- 將現有系統的功能需求整合到 UMS 需求規格中
- 確保 UMS 能提供等同或更優的功能體驗
- 建立標準化的 API 介面，為未來整合做準備

**現有系統功能需求整合**：
- 盤點 AIADJ/廠務 系統的核心功能與工作流程
- 將關鍵功能轉化為 UMS 的功能需求
- 確保新平台能滿足現有使用者的操作習慣與業務需求

### 第二階段：整合與遷移策略
**目標**：將現有系統（AIADJ/廠務）逐步整合並遷移至 UMS

1. **評估階段**
	- 詳細盤點現有系統功能、資料格式、API、工作流程
	- 分析與 UMS 的功能對應關係與差異
2. **設計階段**
	- 定義資料匯出/匯入格式、欄位對應 (mapping)
	- 設計同步或切換策略（雙寫、批次導入或切換式）
3. **遷移執行（分階段）**
	- Pilot：選定小範圍資料與使用者進行遷移測試
	- 全量遷移：分批次導入、驗證資料一致性
	- 切換（Cutover）：在業務可接受的維護窗口完成最終切換
4. **回退與備援**
	- 每階段保留回退機制（確保可回到 AIADJ/廠務）
	- 建立驗證腳本來比對樣本結果與服務可用性

**注意事項**：第二階段將在第一階段 UMS 平台穩定運行並經過充分驗證後才啟動。

## 時程與里程碑（建議預估）

### 第一階段：UMS 平台建立（約 24-32 週）
- **第 0 期** — 啟動與需求確認（2–4 週）：利害關係人工作坊、現有系統功能盤點、UMS 詳細需求、驗收準則
- **第 1 期** — 設計與原型（4–6 週）：架構設計、API 規格、UI 原型
- **第 2 期** — 基礎設施與平台骨幹（6–8 週）：K8s 叢集、物件儲存、DB、認證整合
- **第 3 期** — 核心功能開發（8–12 週）：資料上傳、標註整合、訓練 Job 提交、Model Registry
- **第 4 期** — 測試與驗收（4–6 週）：整合測試、負載測試、安全測試、使用者驗收

### 第二階段：整合與遷移（約 12-16 週，第一階段完成後啟動）
- **第 5 期** — 整合準備（2–4 週）：遷移策略詳細設計、工具開發、測試環境準備
- **第 6 期** — Pilot 遷移（4–6 週）：與 AIADJ/廠務 小量遷移測試、問題修正
- **第 7 期** — 全量遷移（6–10 週）：分批次遷移、驗證、最終切換

**總預估時程**：36-48 週（依人力與優先順序調整）

## 成員與資源建議
- 專案經理（1）
- 平台後端工程（2）
- 前端工程（1）
- ML/資料工程（1）

基礎設施：GPU叢集架構、物件存儲（S3 兼容或公司內部儲存）、GPU 節點（視訓練負載而定）、CI/CD pipeline、監控平台。

## 風險與緩解措施
- 風險：GPU 資源不足或成本超出預算 — 緩解：採階段式擴充、先使用雲端或租用外部 GPU 作為緩衝
- 風險：遷移導致業務中斷 — 緩解：Pilot、雙寫或延遲切換、明確回退流程
- 風險：現有系統介面難以整合 — 緩解：必要時進行現有系統輕量改造，或建立中介服務
- 風險：原有系統使用者抗拒新系統 — 緩解：微調現有工作流程，保持原有介面，透過API串接減少學習成本

## 驗收標準
- 功能：資料上傳、標註、訓練提交流程可執行且無重大錯誤
- 效能：在指定樣本資料集上，訓練任務能成功啟動且資源使用符合預期
- 穩定性：關鍵服務在 1 週內無重大中斷
- 整合：能透過 API 與至少一個外部系統（例如 AIADJ 測試端）完成資料交換
- 遷移驗證：Pilot 資料與目標系統資料一致率達到預先同意門檻（例如 99%）
